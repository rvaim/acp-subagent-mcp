# 实现指南

本文面向刚开始学习编程或刚开始接触 MCP、ACP 的同学，按“先看哪里、再改哪里、如何验证”的顺序解释本项目。

## 1. 先理解项目在做什么

这个项目像一个“任务调度员”。主代理不会直接去启动 Codex、Claude、Gemini 或其他命令行代理，而是调用本 MCP Server 暴露的工具。本 MCP Server 再根据配置文件启动一个支持 ACP 的子代理，把结构化任务发给它，等待结果，最后把精简后的结果返回给主代理。

可以用一个生活类比理解：

```text
用户提出需求
  ↓
主代理决定要不要找帮手
  ↓
本 MCP Server 像项目经理一样分配任务
  ↓
ACP 子代理像具体工程师一样执行任务
  ↓
本 MCP Server 收集结果、整理摘要、记录日志
  ↓
主代理拿到结果后继续决策
```

## 2. 推荐阅读顺序

第一次看代码时，建议按下面顺序：

1. `README.md`：先跑起来。
2. `doc/API.md`：知道主代理能调用哪些工具。
3. `examples/agents.toml`：理解 agent 是怎么配置的。
4. `src/server.ts`：看 MCP tools 是怎么注册的。
5. `src/runtime/subagentRuntime.ts`：看一次任务如何从输入变成结果。
6. `src/acp/AcpClient.ts`：看 MCP Server 如何调用 ACP agent。
7. `src/acp/JsonRpcStdioTransport.ts`：看 JSON-RPC over stdio 如何收发。
8. `src/task/renderSubagentPrompt.ts`：看结构化任务如何变成 prompt。
9. `src/task/parseSubagentResult.ts`：看子代理结果如何解析。
10. `src/runtime/security.ts`：看安全校验。

## 3. 一次 `subagent_run` 的代码流程

下面是同步任务的真实代码流程。

### 第一步：MCP Host 调用工具

MCP Host 调用 `subagent_run`。工具注册在：

```text
src/server.ts
```

该文件会把工具输入交给：

```text
SubagentRuntime.run(input)
```

### 第二步：运行时校验输入

代码位置：

```text
src/runtime/subagentRuntime.ts
```

`prepareRun` 会做这些事：

- 检查 `agent_type` 是否存在于 `agents.toml`。
- 检查 `cwd` 是否在允许根目录里。
- 检查 `task.files[].path` 是否为相对路径。
- 拒绝 `../` 这种路径穿越。
- 检查 inline/snippet 内容长度。
- 创建日志目录。
- 把结构化任务渲染成 prompt。
- 创建任务对象并放入任务注册表。

### 第三步：启动或复用 ACP session

代码位置：

```text
src/runtime/subagentRuntime.ts
src/acp/AcpClient.ts
```

运行时会先看会话池里有没有可以复用的 session。如果没有，就新建 `GenericAcpClient`。

新建 client 时会启动子代理进程：

```text
spawn(command, args)
```

这里的 `command` 和 `args` 只能来自配置文件，不能由工具调用者传入。

### 第四步：完成 ACP 初始化

代码位置：

```text
src/acp/AcpClient.ts
```

发送：

```text
initialize
```

然后发送：

```text
session/new
```

拿到 `sessionId` 后，才可以发送任务。

### 第五步：发送 prompt

代码位置：

```text
src/acp/AcpClient.ts
```

发送：

```text
session/prompt
```

prompt 内容来自：

```text
src/task/renderSubagentPrompt.ts
```

它会把任务标题、目标、文件、约束、成功标准、返回格式要求都整理成稳定的中文说明。

### 第六步：收集 session/update

代码位置：

```text
src/acp/eventAggregator.ts
```

ACP agent 在执行中可能发出 `session/update` 通知，例如文本片段、工具调用、文件修改提示等。聚合器会把这些事件整理成：

- 文本输出
- 工具调用摘要
- 触碰过的文件
- 错误信息

### 第七步：解析最终结果

代码位置：

```text
src/task/parseSubagentResult.ts
```

解析逻辑很宽容：

1. 优先找 JSON 代码块。
2. 找不到时尝试把完整文本当 JSON 解析。
3. 还失败时，把输出当普通文本保存到 `raw_output`。

这样做的原因是：真实 agent 不一定总能严格只输出 JSON。

### 第八步：写日志并返回精简结果

代码位置：

```text
src/runtime/logs.ts
src/runtime/outputCompaction.ts
```

每次运行会生成：

```text
.subagent-runs/run_xxx/
  task.json
  rendered_prompt.md
  events.jsonl
  stderr.log
  result.json
```

工具输出默认只返回摘要、结构化结果和日志路径，避免把很长的原始事件塞回主代理上下文。

## 4. 如何接入一个真实 ACP agent

假设你有一个命令：

```bash
my-agent --acp
```

只要它支持 ACP stdio，就可以在 `agents.toml` 中加入：

```toml
[agents.my_agent]
description = "我的 ACP 子代理"
command = "my-agent"
args = ["--acp"]
capabilities = ["analysis", "review", "edit"]
```

然后用 `subagent_list` 应该能看到：

```json
{
  "name": "my_agent",
  "description": "我的 ACP 子代理",
  "capabilities": ["analysis", "review", "edit"]
}
```

## 5. 如何添加一个新 MCP 工具

新增工具一般分三步。

### 第一步：定义输入输出类型

文件：

```text
src/task/types.ts
```

新增导出的 interface，并加中文 TSDoc 注释。

### 第二步：定义 Zod schema

文件：

```text
src/tools/schemas.ts
```

Zod schema 用来校验 MCP tool 输入。

### 第三步：注册工具

文件：

```text
src/server.ts
```

调用 `server.registerTool(...)` 注册工具，并把 handler 转交给 `SubagentRuntime`。

## 6. 如何修改 prompt 模板

文件：

```text
src/task/renderSubagentPrompt.ts
```

建议修改时遵守三条规则：

1. 不要把父代理完整对话塞进去。
2. 文件 action 要说清楚，例如 `review` 不能修改文件。
3. 返回格式要稳定，最好继续要求 JSON。

## 7. 如何调试失败

先看 MCP 工具返回的：

```json
{
  "raw_event_log_path": ".subagent-runs/run_xxx/events.jsonl"
}
```

然后查看对应日志：

```bash
cat .subagent-runs/run_xxx/result.json
cat .subagent-runs/run_xxx/events.jsonl
cat .subagent-runs/run_xxx/stderr.log
```

常见判断方法：

- `result.json` 有错误：先看错误码。
- `stderr.log` 有内容：说明子代理进程自己报错。
- `events.jsonl` 没有任何事件：可能是子代理没有按 ACP 协议输出 JSON-RPC。
- 发生 `HEARTBEAT_TIMEOUT`：说明指定时间内没有任何存活信号。

## 8. 修改代码后的验证步骤

每次改完代码，至少运行：

```bash
npm run check
npm run build
```

再运行 fake agent 连通性测试：

```bash
node --input-type=module <<'JS'
import { loadConfig } from './dist/config/loadConfig.js';
import { SubagentRuntime } from './dist/runtime/subagentRuntime.js';

const config = await loadConfig('examples/agents.toml');
const runtime = new SubagentRuntime(config);
const output = await runtime.run({
  agent_type: 'fake',
  cwd: process.cwd(),
  task: {
    title: '验证任务',
    goal: '确认代码修改后仍可正常运行',
    expected_output: { format: 'structured' }
  },
  session_pool_policy: 'disable',
  detail_level: 'normal'
});
console.log(JSON.stringify(output, null, 2));
await runtime.shutdown();
JS
```

## 9. 新手最容易踩的坑

### 坑一：`cwd` 不在允许目录里

修改 `agents.toml`：

```toml
[security]
allowed_cwd_roots = ["/你的项目根目录"]
```

### 坑二：子代理不是 ACP agent

普通 CLI 不一定支持 ACP。它必须能收发 JSON-RPC。

### 坑三：输出不是 JSON

没有关系。解析器会降级成普通文本。但如果主代理后续要稳定消费，仍建议让子代理尽量返回 JSON。

### 坑四：心跳超时太短

默认是 60 秒。如果某个 agent 在长工具调用期间会短暂静默，一般不需要调整；如果某个 agent 冷启动或工具执行超过 60 秒没有任何 stdout/stderr/ACP 消息，可以只给该 agent 提高：

```toml
[agents.slow]
heartbeat_timeout_secs = 10
```
