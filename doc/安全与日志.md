# 安全与日志

本文说明本 MCP Server 如何控制风险，以及如何利用日志排查问题。

## 1. 安全目标

本项目会启动本地子进程，并可能让子代理读取或修改文件。因此安全目标是：

- 不允许工具调用者执行任意命令。
- 不允许子代理访问未授权目录。
- 不允许路径穿越。
- 不默认允许写文件或执行命令。
- 不把密钥写入日志。
- 子代理失败、取消或超时时要清理进程。

## 2. agent_type 白名单

工具输入只能传：

```json
{"agent_type":"codex"}
```

`codex` 必须存在于配置：

```toml
[agents.codex]
command = "codex-acp"
args = []
```

工具调用者不能传：

```json
{"command":"rm","args":["-rf","/"]}
```

因为工具 schema 根本不包含这些字段，运行时也只读取配置中的命令。

## 3. cwd 安全校验

`cwd` 必须满足：

- 是绝对路径。
- 真实路径位于 `[security].allowed_cwd_roots` 下。

例如：

```toml
[security]
allowed_cwd_roots = ["/Users/me/projects"]
```

允许：

```text
/Users/me/projects/app1
```

拒绝：

```text
/Users/me/.ssh
/etc
```

代码位置：

```text
src/runtime/security.ts
```

## 4. 文件路径安全校验

任务里的文件路径必须是相对 `cwd` 的路径。

允许：

```json
{"path":"src/index.ts"}
```

拒绝：

```json
{"path":"../secret.txt"}
{"path":"/etc/passwd"}
```

原因是：子代理只应该处理当前任务授权的项目文件。

## 5. inline 和 snippet 限制

`content_mode=inline` 或 `content_mode=snippet` 会把文件内容放进 prompt。为了避免 prompt 爆炸，单个内容不能超过：

```toml
max_inline_file_chars = 30000
```

如果需要处理大文件，推荐使用：

```json
{"content_mode":"path_only"}
```

让 ACP agent 自己在 `cwd` 下读取。

## 6. 权限策略

默认权限建议保守：

```toml
[permissions.default]
read = "allow"
search = "allow"
edit = "deny"
execute = "deny"
network = "deny"
```

如果某个 agent 明确需要改文件，可以单独放宽：

```toml
[permissions.codex]
read = "allow"
search = "allow"
edit = "allow"
execute = "deny"
network = "deny"
```

当 ACP agent 发起权限请求时，本 MCP Server 会根据策略自动返回允许、拒绝或取消。

## 7. 日志目录

每次运行都会创建一个目录：

```text
.subagent-runs/run_20260101_120000_abcd/
```

里面有：

| 文件 | 作用 |
|---|---|
| `task.json` | 脱敏后的结构化任务 |
| `rendered_prompt.md` | 实际发送给子代理的 prompt |
| `events.jsonl` | ACP 原始事件，每行一个 JSON |
| `stderr.log` | 子代理 stderr 输出 |
| `result.json` | 最终结构化结果 |

## 8. 日志脱敏

代码位置：

```text
src/runtime/redact.ts
```

会对常见敏感字段做脱敏，例如：

- token
- secret
- password
- api_key
- access_key
- authorization

脱敏不是绝对安全边界。不要主动把生产密钥写入任务描述或 inline 文件内容。

## 9. 如何用 `subagent_logs` 查看日志

输入示例：

```json
{
  "task_id": "task_abc",
  "kind": "events",
  "max_chars": 12000
}
```

`kind` 可选：

```text
task
prompt
events
stderr
result
```

返回的是日志尾部，避免一次返回太多内容。

## 10. 常见错误码

| 错误码 | 含义 | 排查方向 |
|---|---|---|
| `TASK_NOT_FOUND` | 任务 ID 不存在 | 检查 task_id 是否写错 |
| `START_FAILED` | 任务启动失败 | 看 stderr.log 和配置 command |
| `TIMEOUT` | 总运行时间超时 | 提高 timeout_secs 或优化任务 |
| `INACTIVITY_TIMEOUT` | 长时间没有有效进展 | 看 events.jsonl 是否有活动 |
| `HEARTBEAT_TIMEOUT` | 子代理没有任何存活信号 | 检查 agent 是否按 ACP 输出 |
| `CANCELLED` | 任务被取消 | 看取消来源 |
| `SUBAGENT_FAILED` | 子代理或协议异常 | 看 result.json 和 stderr.log |

## 11. 进程清理

取消、超时或失败时会尝试：

1. 发送 `session/cancel`。
2. 关闭 JSON-RPC transport。
3. 发送 SIGTERM。
4. 必要时发送 SIGKILL。

类 Unix 系统优先清理进程组，降低孤儿进程风险。

## 12. 生产环境建议

- `allowed_cwd_roots` 不要写 `/`。
- 默认不要允许 `execute` 和 `network`。
- 给不同 agent 配不同权限。
- 定期清理 `.subagent-runs`。
- 把 `heartbeat_timeout_secs` 调到适合具体 agent 冷启动的值。
- 真实生产前用 fake agent 和测试仓库先跑通链路。
