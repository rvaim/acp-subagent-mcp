# 原理说明

本文用尽量直观的方式解释本项目背后的几个核心概念：MCP、ACP、JSON-RPC、stdio、心跳、取消、会话池和输出压缩。

## 1. MCP 是什么

在本项目里，可以把 MCP 理解为“主代理调用外部能力的工具协议”。主代理通过 MCP 发现工具、调用工具、拿到工具结果。

本项目暴露的 MCP tools 包括：

```text
subagent_list
subagent_run
subagent_start
subagent_start_many
subagent_wait
subagent_result
subagent_continue
subagent_cancel
subagent_close
subagent_logs
```

主代理只需要关心这些高层工具，不需要知道 ACP 底层怎么初始化、怎么创建 session、怎么读 stdout。

## 2. ACP 是什么

ACP 更像“客户端和 coding agent 之间的会话协议”。一个支持 ACP 的 agent 通常会经历：

```text
initialize
  ↓
session/new
  ↓
session/prompt
  ↓
session/update
  ↓
session/cancel，可选
```

本项目在 ACP 侧扮演 client。也就是说，本 MCP Server 是主代理眼里的工具服务器，同时又是 ACP 子代理眼里的客户端。

## 3. 为什么不能直接把 ACP 方法暴露成 MCP 工具

一种简单但不好的做法是暴露这些工具：

```text
initialize
session_new
session_prompt
session_update
session_cancel
```

这样主代理就要自己管理协议细节，调用步骤很多，也容易出错。

本项目选择暴露高层工具：

```text
subagent_run
subagent_start
subagent_wait
subagent_continue
```

好处是：

- 主代理只描述任务，不操作协议细节。
- MCP Server 可以统一做安全校验。
- MCP Server 可以统一处理超时、取消、日志和会话池。
- 后续换不同 ACP agent 时，主代理工具面保持稳定。

## 4. JSON-RPC over stdio 是什么

JSON-RPC 是一种用 JSON 表示请求、响应和通知的协议。

stdio 是标准输入和标准输出：

- MCP Server 向子代理进程的 stdin 写消息。
- 子代理从 stdin 读消息。
- 子代理向 stdout 写回消息。
- MCP Server 从 stdout 读取响应。

本项目的传输层在：

```text
src/acp/JsonRpcStdioTransport.ts
```

它做了几件事：

- 给每个请求分配 id。
- 把 JSON 对象序列化成一行文本写入 stdin。
- 从 stdout 按行读取 JSON。
- 根据 id 匹配 request 和 response。
- 收到 notification 时交给事件聚合器。
- 收到反向 request 时执行权限或文件系统处理。

## 5. request、response、notification 的区别

### request

需要对方回复。例如：

```json
{"jsonrpc":"2.0","id":1,"method":"initialize","params":{}}
```

### response

对 request 的回复。例如：

```json
{"jsonrpc":"2.0","id":1,"result":{}}
```

### notification

不需要回复。例如：

```json
{"jsonrpc":"2.0","method":"session/update","params":{}}
```

`session/update` 通常用于子代理在执行过程中告诉 client：“我有新进度了”。

## 6. 心跳为什么重要

只有 wall-clock timeout 不够。假设主代理或 MCP Server 退出，但真实 agent 仍在后台跑，就可能留下孤儿进程。

所以本项目加入了父子双向心跳监督。默认规则是：

```text
MCP Server 定期发送 JSON-RPC 心跳请求；子代理包装器立即回复。
MCP Server 收到回复后确认子代理仍可通信。
子代理包装器默认 3 秒收不到主代理心跳，就主动关闭自身和真实 ACP agent。
```

这不是“主代理检测子代理是否输出”的机制。子代理可以在长工具调用、长文本生成或 Claude Write/Bash 期间保持安静，只要 JSON-RPC 心跳仍能往返，就不会因为 heartbeat 被杀。

相关组件：

- `src/runtime/heartbeat.ts`：发送主代理心跳请求，并监控子代理心跳回复。
- `src/runtime/processManager.ts`：启动子代理包装器，由包装器拦截 `$/subagent_mcp/heartbeat`、回复心跳，并在主代理失联时终止进程组。


## 7. inactivity timeout 和 heartbeat timeout 的区别

| 类型 | 看什么 | 默认值 | 含义 |
|---|---|---:|---|
| heartbeat timeout | 子代理包装器是否还能读到主代理心跳 | 60 秒 | 主代理/MCP Server 可能已退出或卡死 |
| inactivity timeout | 子代理是否有有效工作进展 | 120 秒 | 子代理可能没有继续推进任务 |

一个 agent 可以长时间没有 stdout/stderr，但只要主代理心跳仍在更新，heartbeat 不会触发。真正没有有效工作进展时由 `inactivity_timeout_secs` 处理。

## 8. 为什么要写日志

MCP tool output 不适合返回完整日志。原因是完整日志可能非常长，会浪费主代理上下文。

所以本项目采用：

```text
短结果返回给主代理
完整事件写入本地文件
```

日志目录结构：

```text
.subagent-runs/run_xxx/
  task.json
  rendered_prompt.md
  events.jsonl
  stderr.log
  result.json
```

这样主代理平时只看摘要，需要排查时再调用 `subagent_logs` 或直接看文件。

## 9. 会话池原理

每次启动 ACP agent 都有成本：进程启动、initialize、session/new、模型或运行时缓存准备。

会话池的思想是：

```text
一个任务完成后，如果 session 还可用，就先放进池子。
下次同一个主代理、同一种 agent、同一个 cwd、同一种模式的任务来了，就复用它。
```

但复用有风险，所以必须满足兼容条件：

- `parent_agent_id` 相同，防止不同主代理串上下文。
- `agent_type` 相同。
- `cwd` 相同。
- `mode` 相同。
- `mcp_servers` 签名相同。
- 权限策略签名相同。
- session 空闲。
- 上下文使用率低于 90%。

代码位置：

```text
src/runtime/sessionPool.ts
```

## 10. 输出压缩原理

不同详情级别对应不同信息量：

```text
summary：只返回主代理必须消费的信息
normal：多返回工具调用摘要和更多结果
verbose：尽量保留更多调试信息
```

代码位置：

```text
src/runtime/outputCompaction.ts
```

这样做是为了减少主代理和子代理之间的 token 消耗。

## 11. 取消和进程清理原理

取消时只发 `session/cancel` 不够，因为子代理进程可能不理会取消通知。

本项目采用多级清理：

```text
MCP notifications/cancelled / subagent_cancel / timeout / heartbeat timeout / stdio 断开
  ↓
标记任务为 cancelled、timeout 或 heartbeat_timeout
  ↓
发送 ACP session/cancel
  ↓
关闭 transport
  ↓
发送 SIGTERM
  ↓
按进程组和父子关系补杀后代进程
  ↓
短宽限期后补发 SIGKILL
```

类 Unix 系统上，子进程以 detached 方式启动，清理时优先杀整个进程组，避免留下孙进程。由于 Claude 这类 CLI 可能通过 shell、npx、node 或 pty 再拉起真实工作进程，代码还会读取 `ps -eo pid=,ppid=`，按父子关系补杀后代进程。Windows 下使用 `taskkill /T` 清理进程树。

心跳和取消不是同一个机制：心跳负责发现“没有任何存活响应”的 agent；用户点击停止当前 MCP tool call 时，必须监听 MCP SDK 提供的 AbortSignal，并立即走取消清理链路。

代码位置：

```text
src/runtime/processManager.ts
```

## 12. 安全边界原理

本项目把“能运行什么命令”和“能访问什么目录”都放在配置里，而不是让工具调用者决定。

核心原则：

- `agent_type` 必须来自配置文件。
- 工具输入不能传 `command` 和 `args`。
- `cwd` 必须在允许根目录下。
- 文件路径必须是相对路径。
- 文件路径不能逃逸 `cwd`。
- inline 内容有长度限制。
- 日志会脱敏 token、secret、key 等敏感信息。

代码位置：

```text
src/runtime/security.ts
src/runtime/redact.ts
```
