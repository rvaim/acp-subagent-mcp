# 常见问题

## 1. 启动时报 `找不到 agents.toml`

原因：没有设置配置路径，或者当前目录没有 `agents.toml`。

解决：

```bash
cp examples/agents.toml agents.toml
SUBAGENT_MCP_CONFIG=./agents.toml node dist/index.js
```

## 2. `cwd 必须位于 allowed_cwd_roots 下`

原因：工具输入中的 `cwd` 不在配置允许的根目录里。

解决：修改配置：

```toml
[security]
allowed_cwd_roots = ["/你的项目目录"]
```

不要为了省事写 `/`，那会放大风险。

## 3. `未知 agent_type`

原因：工具输入里的 `agent_type` 没有对应配置。

解决：检查 `agents.toml` 是否有：

```toml
[agents.xxx]
```

如果配置名是 `[agents.fake]`，工具里就要传：

```json
{"agent_type":"fake"}
```

## 4. fake agent 可以跑，真实 agent 不行

优先检查：

1. 真实 agent 是否支持 ACP stdio。
2. `command` 是否在 PATH 里。
3. `args` 是否正确。
4. `stderr.log` 是否有报错。
5. `events.jsonl` 是否有 JSON-RPC 消息。

## 5. 出现 `HEARTBEAT_TIMEOUT`

含义：在 `heartbeat_timeout_secs` 内没有收到任何存活信号。

常见原因：

- 子代理启动太慢。
- 子代理没有按 ACP 协议向 stdout 输出 JSON-RPC。
- 子代理把日志写到了 stdout，污染了 JSON-RPC 通道。
- 子代理卡死。

解决：

- 先看 `stderr.log`。
- 确认子代理 stdout 只输出协议消息。
- 对冷启动慢的 agent 单独调高：

```toml
[agents.slow_agent]
heartbeat_timeout_secs = 10
```

## 6. 出现 `INACTIVITY_TIMEOUT`

含义：子代理可能还活着，但长时间没有有效工作进展。

解决：

- 看 `events.jsonl` 是否持续有心跳但没有任务输出。
- 拆小任务。
- 提高 `inactivity_timeout_secs`。

## 7. 为什么 `subagent_run` 默认不返回完整 events

因为完整事件可能很长，会污染主代理上下文。

本项目默认只返回摘要和结构化结果。需要排查时，可以调用：

```json
{"task_id":"task_xxx","kind":"events"}
```

或者直接打开：

```text
.subagent-runs/run_xxx/events.jsonl
```

## 8. 子代理返回的不是 JSON 会失败吗

不会。解析器会降级为普通文本结果。

不过如果你希望主代理稳定判断成功、风险、文件改动，最好让子代理返回结构化 JSON。

## 9. 为什么 `subagent_start` 返回的 `session_id` 可能为空

`subagent_start` 会立即返回 `task_id`，实际 ACP 初始化在后台进行。如果返回瞬间 session 还没创建好，`session_id` 就可能为空。

后续可以用：

```text
subagent_result
subagent_wait
```

查询任务状态和最终结果。

## 10. `subagent_continue` 报上一轮还没结束

同一个 ACP session 同一时间只允许一个 active prompt turn。

解决：先调用 `subagent_wait` 等上一轮完成，再调用 `subagent_continue`。

## 11. 为什么写任务不能随便并行

多个子代理同时写同一个项目目录容易互相覆盖。

默认策略是：

```toml
default_conflict_policy = "single_writer_per_cwd"
```

意思是同一个 `cwd` 同时只允许一个写任务。

## 12. `sandbox_worktree` 能直接用吗

目前不能。它只是配置预留，尚未实现真实 git worktree 隔离、diff 提取和合并。

## 13. 如何清理旧日志

可以直接删除日志目录：

```bash
rm -rf .subagent-runs
```

注意：删除后旧任务的日志就无法查看了。

## 14. 如何确认代码没有 TypeScript 错误

运行：

```bash
npm run check
```

如果没有输出 TypeScript 错误，就表示类型检查通过。

## 15. 如何重新编译

运行：

```bash
npm run build
```

编译产物在：

```text
dist/
```
